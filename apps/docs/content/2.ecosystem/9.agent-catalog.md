# Agent Catalog

The OpenApe ecosystem provides 29 predefined personas that the Owner can spawn to staff a virtual company. Each persona maps to a pinned recipe in the [agent-catalog](https://github.com/openape-ai/agent-catalog) and occupies a structural role in the ORG chart.

## Persona Categories

Personas are grouped into 7 categories:

| Category | Label |
|----------|-------|
| `leadership` | Leadership & Coordination |
| `engineering` | Engineering |
| `design-content` | Design & Content |
| `data-research` | Data & Research |
| `growth-sales` | Growth & Sales |
| `operations` | Operations & Support |
| `finance-legal` | Finance & Legal |

## Tool Catalog

The runtime ships with 21 built-in tools that agents can use. The UI's tool-picker fetches this list via `GET /api/tool-catalog`.

| Tool | Risk | Description |
|------|------|-------------|
| `time.now` | low | Current UTC timestamp + ISO date. |
| `http.get` | medium | GET an HTTPS URL and return the body (capped at 1MB). |
| `http.post` | medium | POST JSON to an HTTPS URL and return the response body. |
| `file.read` | medium | Read a file from the agent's home directory ($HOME). |
| `file.write` | high | Write a file in the agent's home directory ($HOME). |
| `tasks.list` | low | List the owner's open ape-tasks, optionally filtered by status. |
| `tasks.create` | medium | Create a new ape-task on the owner's task list. |
| `mail.list` | low | List recent inbox messages via o365-cli. |
| `mail.search` | low | Search inbox via o365-cli with a query string. |
| `bash` | high | Run a shell command on the agent host via ape-shell. Every command goes through the DDISA grant cycle — auto-approved by a matching YOLO scope or owner push notification. |
| `file.edit` | low | Replace an exact substring in a file under the agent's home directory. Touches only the changed region. |
| `git.worktree` | high | Manage isolated git worktrees for coding tasks: create, remove, list. Git operations go through the DDISA grant cycle. |
| `verify` | medium | Run the verification command (tests/build/lint) in a worktree and report pass/fail. |
| `forge.pr.create` | high | Open a pull request on GitHub or Azure DevOps. |
| `forge.pr.merge` | high | Merge a PR or arm merge-when-green (auto). Never bypasses required checks. |
| `forge.pr.status` | low | Fetch a PR's state, checks, and review decision. |
| `forge.issue.get` | low | Fetch an issue (GitHub) or work-item (Azure) — title, body, labels. |
| `agent.spawn` | high | Spawn a worker agent on the nest via troop. Requires the troop:spawn-agent scope. |
| `agent.destroy` | high | Destroy a worker agent on the nest (full teardown). Requires the troop:destroy-agent scope. |
| `troop.company.read` | low | Read your troop company (objectives | reports | members | cost-snapshots | overview). Read-only. |
| `troop.objective.upsert` | medium | Create or update a company objective on troop.openape.ai. |

> **Note:** Tools marked `high` risk go through the DDISA grant cycle and require owner approval or a matching YOLO scope. Tools marked `medium` risk may also require approval depending on the context.

## ORG Composition

The Owner spawns personas from the org chart using the recipe parameters. Each persona has its own set of parameters:

- **Common parameters**: `org_id`, `org_name` (substituted at spawn time)
- **Coding personas**: Also include `forge_base: "https://git.openape.ai"` for Forgejo interaction

The `forge_base` parameter is included for personas with `coding: true` that need to interact with Forgejo (git.openape.ai).

## Spawn Process

1. Owner views the org chart in the troop UI
2. Owner clicks "Spawn" on an available persona slot
3. The system substitutes `{{org_id}}` and `{{org_name}}` with the company's values
4. The agent is deployed with the pinned recipe version
5. The agent appears in the team members list with its assigned role
